# keenKonnect — Kintsugi v1 (Hybrid)

**Open Builder Stack “under one roof” (Annex + Mimic)**

**Document status:** Draft (v1)
**Date:** 2026-01-28
**Applies to:** keenKonnect (Konstruct + Stockage), **Kintsugi track** (not Kompendio)

---

## 1) Executive summary

In keenKonnect, **Kintsugi** is the plan to bring a *coherent open-source builder stack* under one roof for **hybrid** deployments (SaaS + self-host). It does this by:

* **Annexing** open-source components as *sidecars* (deployable services) and integrating them via standard contracts.
* **Mimicking** useful UX/workflow patterns when annexing would create licensing or product-coherence risk.

**Kompendio is separate**: Kompendio is the repertory of reference sites/charts; **Kintsugi is the integrated open-source toolchain**.

---

## 2) Goals and non-goals

### Goals (v1)

1. Provide a **default hybrid-safe open-source stack** for build execution and artifact handling.
2. Make it feel like **one product** via:

   * SSO (OIDC)
   * canonical IDs
   * unified permissions (RBAC)
   * common activity/events
   * exportable “release packs”
3. Allow **swapability** per deployment (self-host customers can replace components).

### Non-goals (v1)

* Replace proprietary platforms (e.g., Onshape/McMaster) with full mirrors.
* Turn keenKonnect into a monolith by embedding large platforms into core.
* Build every connector; focus on a small set with clean contracts.

---

## 3) Hybrid constraints (licensing posture)

Hybrid means you must assume:

* You will **host** services for users (network interaction).
* You will also ship **self-host** deployments.
* You want **minimal copyleft friction by default**, with **optional copyleft sidecars** when acceptable.

AGPL is explicitly designed to address “network server software” scenarios. ([GNU][1])
MinIO’s own licensing page describes a dual-license model (AGPLv3 or commercial) and states obligations when hosting/deriving over a network. ([MinIO][2])

**Policy (v1):**

* Default stack: permissive licenses (Apache/MIT) where possible.
* Copyleft/AGPL components: **optional**, isolated as sidecars, with an explicit “license mode” switch per deployment.

*(Not legal advice; treat as product policy + engineering constraints.)*

---

## 4) “Under one roof” integration contract (required for every annexed component)

Each annexed service must implement these five contracts:

1. **Identity:** OIDC SSO and RBAC mapping (groups/roles).
2. **Canonical ID mapping:** every external object (doc, BOM item, release, etc.) maps to a Konnaxion/keenKonnect canonical ID (no dual truth).
3. **Events:** webhooks → keenKonnect activity stream (project timeline, audit trail).
4. **Artifact contract:** any important output can be packaged and stored as a versioned artifact in Stockage.
5. **Exit/portability:** exportable dataset or pack format (minimum: metadata + manifests + files).

---

## 5) Stack architecture (lanes)

Kintsugi is organized as **lanes**. Each lane has a default OSS choice + optional alternatives.

### Lane A — Inventory/BOM (open “parts DB” backbone)

**Default:** InvenTree (MIT). ([PyPI][3])
Why: API-friendly inventory + BOM backbone without AGPL network-copyleft.

**Optional (AGPL):** Part-DB (AGPLv3). ([GitHub][4])
Use only if deployment accepts AGPL obligations and you keep it isolated.

### Lane B — Search & indexing (cross-project discovery)

**Default:** OpenSearch (Apache 2.0). ([OpenSearch][5])

### Lane C — Collaborative docs (specs, build logs, checklists)

**Default:** Etherpad (Apache 2.0). ([GitHub][6])
**Optional (AGPL):** HedgeDoc (AGPLv3). ([GitHub][7])

### Lane D — Object storage (Stockage backend for large binaries)

**Default:** SeaweedFS (Apache 2.0). ([GitHub][8])
**Optional:** Ceph (most of Ceph is dual-licensed LGPL 2.1/3.0; plus some mixed licensing in parts). ([GitHub][9])

**Optional (AGPL/commercial):** MinIO dual-licensed (AGPLv3 or commercial). ([MinIO][2])

### Lane E — Forge (versioned assets, templates, scripts)

**Default:** Gitea (MIT). ([GitHub][10])
**Optional (GPL):** Forgejo (GPL v3+). ([forgejo.org][11])

### Lane F — CAD/BIM toolchain (export lane, not UI takeover)

**Default approach:** treat CAD/BIM tools as **toolchain lanes** that produce exports (STEP/IFC/etc.) → Stockage → project context.
*(Component selection here can stay flexible; keep the “export contract” consistent.)*

### Lane G — Kubernetes operator layer (if you standardize on K8s)

**Optional:** Rook (Apache 2.0) to manage storage backends. ([Rook][12])

## 6) Default hybrid bundle (v1)

This is the “ship it” set that minimizes licensing friction:

* **InvenTree** (Inventory/BOM) — MIT ([PyPI][3])
* **OpenSearch** (Search) — Apache 2.0 ([OpenSearch][5])
* **Etherpad** (Collab docs) — Apache 2.0 ([GitHub][6])
* **SeaweedFS** (Object storage) — Apache 2.0 ([GitHub][8])
* **Gitea** (Forge) — MIT ([GitHub][10])

**Optional “copyleft mode” add-ons:**

* HedgeDoc (AGPL) ([GitHub][7])
* Part-DB (AGPL) ([GitHub][4])
* MinIO (AGPL/commercial) ([MinIO][2])
* Forgejo (GPL v3+) ([forgejo.org][11])

---

## 7) Data flow (how it works end-to-end)

1. **Project created (Konstruct)** → roles/permissions set.
2. **Docs** created/linked (Etherpad/HedgeDoc) → metadata stored in keenKonnect → snapshots stored in Stockage.
3. **BOM/parts** managed (InvenTree/Part-DB) → BOM snapshots stored in Stockage → referenced by canonical IDs.
4. **Design exports** (CAD/BIM toolchain) → exports stored in Stockage → linked to tasks/releases.
5. **Search** indexes metadata across projects, docs, BOM, charts.
6. **Release pack** compiled: manifests + files + references → stored in Stockage → optionally exported.

## 8) Implementation plan (v1)

### Phase 0 — Contracts and scaffolding

* OIDC SSO integration pattern
* canonical ID mapping service
* webhook/event ingestion into project activity stream
* Stockage artifact contract (manifests + versioning rules)

### Phase 1 — Default bundle integration

* Deploy: InvenTree, OpenSearch, Etherpad, SeaweedFS, Gitea
* Build connectors: auth + events + minimal CRUD bridges
* UI: “one roof” navigation + permissions + audit surface

### Phase 2 — Release packs

* Define pack format (manifest, hashes, references)
* “Pin to release” workflow (BOM snapshot + doc snapshot + exported assets)

### Phase 3 — Optional copyleft mode

* Add toggles per deployment:

  * HedgeDoc vs Etherpad
  * Part-DB vs InvenTree (or both)
  * MinIO vs SeaweedFS/Ceph
  * Forgejo vs Gitea

---

## 9) Acceptance criteria (v1 “done”)

* A deployment can run in **SaaS** and **self-host** with the same product behavior.
* All annexed components:

  * authenticate via SSO
  * emit events into the project timeline
  * store artifacts into Stockage via a stable contract
* BOM, docs, and released artifacts are **versioned** and **recoverable** via Stockage.
* The default bundle uses permissive licensing by default; copyleft components are isolated and explicitly enabled. ([GNU][1])

---

## 10) Risks and mitigations

* **License surprises in hybrid:** keep a “license mode” switch and isolate copyleft services. ([GNU][1])
* **Dual truth stores:** enforce canonical ID mapping and snapshotting rules.
* **UX fragmentation:** enforce the integration contract; the user should experience one product.
* **Ops complexity:** provide a reference Helm/K8s deployment path; Rook optional if K8s-first. ([Rook][12])
