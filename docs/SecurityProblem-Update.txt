You should treat this as **incident recovery + rebuild**, not just “add security.” Your own notes say the attacker reached the `deploy` user, launched malicious Docker containers, added cron persistence, attempted a sudo backdoor user, created `/tmp/sshd`, and exposed secrets. That means the current VPS cannot be trusted long-term. 

## Priority 1 — Stop trusting the current VPS

Do **not** keep hardening the same server as the final fix.

Your best path is:

1. Create a **new VPS**.
2. Do **not clone the old disk**.
3. Deploy only from clean Git/source.
4. Restore only a verified DB dump and required media.
5. Rotate every secret.
6. Lock SSH and firewall before exposing the app.

This matches the recommendation in your incident notes: migrate to a fresh VPS, avoid cloning the old server, rotate secrets, use SSH keys only, disable password login, add cloud firewall + UFW, stop exposing port `3000`, and enable backups/snapshots.

## Priority 3 — Firewall before app deployment

On the **new VPS**, expose only:

```text
22/tcp    SSH, ideally only from your IP
80/tcp    HTTP
443/tcp   HTTPS
```

Do **not** expose:

```text
3000/tcp  Next.js direct access
5555/tcp  Traefik dashboard
5432/tcp  Postgres
6379/tcp  Redis
8000/tcp  Django/Gunicorn
Docker daemon TCP ports
```

Your current guide says Traefik exposes `80/443/5555`, Next.js responds on `3000`, and Django/Postgres/Redis run in Docker. Those are exactly the ports to tighten: public users should reach only Traefik on `80/443`; everything else should be private. 

Use this baseline:

```bash
sudo apt update && sudo apt upgrade -y
sudo apt install -y ufw fail2ban unattended-upgrades

sudo ufw default deny incoming
sudo ufw default allow outgoing

# Replace with your real home/office IP if possible
sudo ufw allow from <YOUR_PUBLIC_IP>/32 to any port 22 proto tcp

# If your IP changes often, temporarily use this instead:
# sudo ufw allow 22/tcp

sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

sudo ufw deny 3000/tcp
sudo ufw deny 5555/tcp
sudo ufw deny 5432/tcp
sudo ufw deny 6379/tcp

sudo ufw enable
sudo ufw status verbose
```

Fail2Ban is useful because it watches logs such as SSH auth logs and bans IPs with repeated failed authentication attempts by updating firewall rules. ([GitHub][1])

## Priority 4 — Lock SSH

Create a new SSH key on your local machine:

```powershell
ssh-keygen -t ed25519 -a 100 -C "konnaxion-new-vps-2026"
```

On the VPS:

```bash
sudo adduser deploy
sudo mkdir -p /home/deploy/.ssh
sudo nano /home/deploy/.ssh/authorized_keys
sudo chown -R deploy:deploy /home/deploy/.ssh
sudo chmod 700 /home/deploy/.ssh
sudo chmod 600 /home/deploy/.ssh/authorized_keys
```

Then harden SSH:

```bash
sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.backup.$(date +%Y%m%d_%H%M%S)

sudo tee /etc/ssh/sshd_config.d/99-konnaxion-hardening.conf >/dev/null <<'EOF'
PermitRootLogin no
PasswordAuthentication no
KbdInteractiveAuthentication no
PubkeyAuthentication yes
AllowUsers deploy
X11Forwarding no
AllowTcpForwarding no
ClientAliveInterval 300
ClientAliveCountMax 2
MaxAuthTries 3
EOF

sudo sshd -t
sudo systemctl reload ssh
```

Keep your current SSH session open. Open a second terminal and test:

```powershell
ssh deploy@NEW_SERVER_IP
```

OpenSSH’s configuration supports disabling password authentication and keyboard-interactive authentication; disabling both matters because keyboard-interactive/PAM can otherwise serve a similar role to password auth. ([man7.org][2])

## Priority 5 — Do not let `deploy` control Docker freely

This is important because the previous breach used Docker containers. Your notes show malicious containers and image `negoroo/amco:123`, and the attacker used cron to restart those containers. 

Avoid this pattern:

```bash
sudo usermod -aG docker deploy
```

Docker’s own docs warn that the `docker` group grants root-level privileges, and Docker says the daemon normally requires root privileges unless using rootless mode. ([Docker Documentation][3])

Better options:

### Simple safer option

Use `deploy` for files, but require sudo password for Docker commands:

```bash
sudo deluser deploy docker || true
sudo gpasswd -d deploy docker || true
```

Then deploy with:

```bash
sudo docker compose -f docker-compose.production.yml up -d --build
```

### Stronger option

Use a separate admin user for deployment, and make the app files owned by a non-sudo user:

```text
ops     = can sudo, used only for maintenance/deploy
deploy  = owns app files, cannot sudo, not in docker group
```

### Best option later

Move deployment to a CI/CD runner with limited SSH access, or use Docker rootless mode. Docker rootless mode runs the Docker daemon and containers as a non-root user to reduce the impact of daemon/container runtime vulnerabilities. ([Docker Documentation][4])

## Priority 8 — Safer deployment flow

Use this deployment model:

```text
Local machine:
1. Build and test.
2. Create clean Git archive.
3. Upload archive.

New VPS:
1. Extract into fresh release directory.
2. Restore only clean env files.
3. Build containers.
4. Run migrations.
5. Start services.
6. Validate only through 80/443.
```

Your guide already recommends a clean `git archive` deployment and warns not to commit deployment archives such as `.tar.gz` or `.zip`. 

Use versioned releases instead of replacing the same directory blindly:

```bash
sudo mkdir -p /opt/konnaxion/releases /opt/konnaxion/shared
sudo chown -R deploy:deploy /opt/konnaxion

RELEASE=$(date +%Y%m%d_%H%M%S)
mkdir -p /opt/konnaxion/releases/$RELEASE
tar -xzf konnaxion-deploy.tar.gz -C /opt/konnaxion/releases/$RELEASE

ln -sfn /opt/konnaxion/releases/$RELEASE /opt/konnaxion/current
```

Store env files outside the release:

```text
/opt/konnaxion/shared/.envs/.production/.django
/opt/konnaxion/shared/.envs/.production/.postgres
/opt/konnaxion/shared/frontend/.env.production
```

Then symlink them into the current release.

## Priority 10 — Backups that do not preserve malware

Use backups for:

```text

```

Do not back up/restore:

```text
entire old disk
/tmp
/dev/shm
crontabs from old server
unknown systemd services

```

Postgres backup example:

```bash
cd /home/deploy/apps/Konnaxion/backend

docker compose -f docker-compose.production.yml exec postgres \
  pg_dump -U konnaxion -d konnaxion \
  > /home/deploy/backups/konnaxion_$(date +%Y%m%d_%H%M%S).sql
```

Then copy the dump off-server:

```powershell
scp deploy@OLD_SERVER_IP:/home/deploy/backups/konnaxion_YYYYMMDD_HHMMSS.sql .
```

## Recommended final target

For production, aim for this:

```text
Cloud firewall:
  allow 80/443 from world
  allow 22 only from your IP
  deny everything else

VPS:
  SSH key only
  root login disabled
  password login disabled
  UFW enabled
  Fail2Ban enabled
  unattended security updates enabled

App:
  Traefik only public entrypoint
  Next.js not public on 3000
  Django not public directly
  Postgres not public
  Redis not public
  Traefik dashboard disabled or private
  secrets rotated
  env files not committed or pasted

Docker:
  deploy user not casually in docker group
  no remote Docker TCP socket
  trusted images only
  regular image/container audit

Recovery:
  fresh VPS
  clean Git deploy
  DB dump only
  media only
  backups/snapshots enabled
```

The most important change is this: **do not try to save the old VPS as your trusted production server.** Use it only as a temporary source for DB/media extraction, then retire it.
